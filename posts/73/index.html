Web API的设计与开发笔记

发表于 2022-09-05 | 分类于 开发

什么是 Web API

Web API 的重要性

  • 通过 API 才能使用的在线服务出现
  • 移动应用与 API
  • API 的经济学

各种各样的 API 模式

  • 将已发布的 Web 在线服务的数据或功能通过 API 公开
  • 将附加在其他网页上的微件公开：例如 Amazon 提供的广告功能
  • 构建现代 Web 应用：单页面应用异步加载
  • 开发智能手机应用
  • 开发社交游戏
  • 公司内部多个系统的集成

应该通过 API 公开什么

  • 公开所有能公开的
  • 公开 API 不会有风险 即使不公开也可以使用爬虫抓取数据
  • 公开 API 能得到什么

设计优美的 Web API 的重要性

  • 设计优美的 Web API 易于使用
  • 设计优美的 Web API 便于更改
  • 设计优美的 Web API 健壮性好
  • 设计优美的 Web API 不怕公之于众

如何美化 Web API

  • 设计规范明确的内容必须遵守相关规范
  • 没有设计规范的内容必须遵守相关事实标准

REST 与 Web API

作为目标对象的开发人员数量与 API 的设计思想

  • LSUD（Large Set of Unknown Developers）：尽可能设计的普适与通用
  • SSKD（Small Set of Known Developers）：易于上手

端点的设计与请求的形式

设计通过 API 公开的功能

SNS聊天功能为例

  • 用户注册与编辑
  • 搜索，添加，删除好友
  • 好友之间的消息交互

API 端点的设计思想

端口的设计

容易记忆，URI 包含的功能一目了然

短小便于输入的 URI

  • http://api.example.com/service/api/search
  • http://api.example.com/search

人可以读懂的 URI

  • http://api.example.com/sv/u
  • http://api.example.com/products/12345

没有大小写的混用的 URI
修改方便的 URI
不会暴露服务端架构的 URI

  • http://api.example.com/cgi-bin/get_user.php?user=100

规则统一的 URI

HTTP 方法与端口

GET获取资源

POST新增资源

PUT更新已有资源

DELETE删除资源

PATCH更新资源的部分属性

HEAD获取资源的元信息

在不支持其他有些 HTTP 方法时解决方法

  • 使用 _method参数
  • 使用 X-Http-Method_Override 头部
  • 请求时统一使用 POST 方法

API 端点的设计

基本设计

  • 获取用户信息列表 GET http://api.example.com/v1/users
  • 新用户注册 POST http://api.example.com/v1/users
  • 更新用户信息 PUT/PATCH http://api.example.com/v1/users/:id
  • 删除用户信息 DELETE http://api.example.com/v1/users/:id

好友关系

  • 获取当前用户的好友列表 GET http://api.example.com/v1/users/:id/friends
  • 添加好友 POST http://api.example.com/v1/users/:id/friends
  • 删除好友 DELETE http://api.example.com/v1/users/:id/friends/:id

动态相关的端点

  • 编辑动态信息 PUT http://api.example.com/v1/updates/:id
  • 删除动态信息 DELETE http://api.example.com/v1/updates/:id
  • 发表动态信息 POST http://api.example.com/v1/updates
  • 获取特定用户的动态信息 GET http://api.example.com/v1/users/:id/updates
  • 获取好友的动态列表 GET http://api.example.com/v1/users/:id/friends/updates

访问资源的端点设计的注意事项

  • 使用名词复数形式
  • 注意所用的单词
  • 不使用空格及需要编码的字符
  • 使用连接符’_’或者’-‘来连接多个单词

搜索与查询参数的设计

获取数据量与获取位置的查询参数

  • limit 与 offset
  • per_page 与 page
  • since_id

参数过滤

  • http://api.example.com/v1/users?name=ken
  • http://api.example.com/v1/users?q=ken

响应数据的设计

数据格式

LSUD：JSON和XML

SSKD高效序列化方式，例如：thrift、protobuf、messagepack

数据格式指定方法

  • URI添加查询参数 format=json
  • 在请求头指定媒体类型 Accept: application/json

使用JSONP

不一定非要实现JSONP，可能会有安全问题
支持JSONP的操作方法 查询参数 callback
JSONP与错误处理

  • 发生了错误依然需要返回200状态码
  • 在响应的数据中添加真正的响应码

数据内部结构的思考方法

让用户来选择响应的内容

  • 查询参数 fields=name,age
  • 指定响应群
    • small
    • medium
    • large

封装是否必要，一般情况都是否
数据是否应该扁平化

  • 层级
  • 扁平化
  • 只有遇到层级结构有绝对优势时才使用层级

序列与格式推荐使用对象来封装数据（如果返回是列表，再在外面加一层对像封装）

  • 更容易理解响应数据表示什么
  • 响应数据通过对像的封装实现了结构统一
  • 可以避免安全方面的风险

该如何返回序列的个数以及是否还有后续数据

  • 服务端要返回20条数据时，可以试着获取21条数据，如果能获取成功，说明至少存在1条后续数据
  • 在返回给客户端的对像中添加还有后续的字段标识 hasNext: true

各个数据的格式

各个数据的名称推荐使用蛇形法
描述性别使用 gender值为male和female

日期格式一般采用RFC 3339 例如：2015-10-12T11:30:22+09:00
大整数时JSON同时传输id和id_str

响应数据的设计

API没有必要如实反映DB的数据表结构

出错信息的表示

通过状态码来表示出错信息

  • 1字头 消息
  • 2字头 成功
  • 3字头 重定向
  • 4字头 客户端原因引起的错误
  • 5字头 服务端原因引起的错误
  • 当找不到合适的状态码的时候，使用’200’，’400’，’500’这些以’00’结尾的状态码

向客户端返回详细的出错信息，把详细的出错信息放入消息体

  • code 内部定义的错误码
  • message 出错信息
  • info 帮助文档的URL

如何填写详细的出错信息？，可以同时提供面向开发人员和非开发人员的信息提示

  • developerMessage
  • userMessage
  • code
  • info

发生错误时防止返回HTML，某些API在发生错误时会把HTML信息写入消息体，尤其是发生500，503以及404等错误时

维护与状态码

  • 当维护时应该返回503状态码
  • 如果能够预测服务何时能够重启可以添加 Retry-After 头

需要返回意义不明确的信息

  • 有时出于安全考虑或其它原因需要返回模棱两可的信息
  • 比如当用户被屏蔽时，如果访问可以返回404，如果返回403对方可能就知道是被屏蔽了
  • 用户登录时，如果明确指出哪一个字段错误，可能会给恶意用户破解的机会

最大程度地利用 HTTP 协议规范

正确使用状态码

1字头 消息
2字头 成功

  • 200 OK 请求成功
  • 201 Created 请求成功，新的资源已创建
  • 202 Accepted 请求成功，请求的操作已接受
  • 204 No Content 没有内容

3字头 重定向

  • 300 Multiple Choices 存在多个资源
  • 301 Moved Permanently 资源被永久转移
  • 302 Found 请求的资源被暂时转移
  • 303 See Other 引用他处
  • 304 Not Modified 自上一次访问后没有发生更新
  • 307 Temporary Redirect 请求的资源被暂时转移

4字头 客户端原因引起的错误

  • 400 Bad Request 请求不正确，一般是数据格式不正确
  • 401 Unauthorized 需要认证
  • 403 Forbidden 禁止访问，一般是没有权限
  • 404 Not Found 没有找到指定的资源
  • 405 Method Not Allowed 没有找到指定的方法
  • 406 Not Acceptable 同Accept相关联的首部里含有无法处理的内容
  • 408 Request Timeout 请求在规定时间内没有处理结束
  • 409 Conflict 资源存在冲突
  • 410 Gone 指定的资源已不存在
  • 413 Request Entity Too Large 请求消息体太大
  • 414 Request-URI Too Long 请求的URI太长
  • 415 Unsupported Media Type 不支持所指定的媒体类型
  • 429 Too Many Requests 请求次数过多，一般用于API请求速度控制

5字头 服务端原因引起的错误

  • 500 Internal Server Error 服务器端发生错误
  • 503 Service Unavailable 服务器暂时停止运行

缓存与HTTP协议规范

过期模型

  • Cache-Control有更多控制方法
  • Expires使用绝对时间来表示到期

验证模型可以使用Last-Modified和Etag

启发式过期，当服务器端没有给出过期时间时，客户端可以决定大约缓存数据多久

不希望实施缓存的情况Cache-Control: no-cache

使用Vary来指定缓存单位

  • 在实施缓存时可能需要同时指定Vary首部
  • Vary用于指定除URI外使用哪个请求首部来确定唯一的数据
  • Vary: Accept-Encoding,User-Agent,Accept-Language

Cache-Control首部

  • public 代理服务器处保存的缓存可以在不同用户之间共享
  • private 每个用户的缓存数据必须各不相同
  • no-cache 缓存数据需要通过验证模式来确认
  • no-store 不需要进行缓存
  • no-transform 代理服务器不可变更响应数据的媒体类型或其他相关内容
  • must-revalidate 不管何时都需要向原始服务器再次验证
  • proxy-revalidate 代理服务器需要向原始服务器再次验证
  • max-age 表示缓存数据处于新鲜状态的时间
  • s-maxage 和max-age一样，但只用于中继服务器
  • stable-while-revalidate 指定的max-age过期后，缓存会异步刷新，但此时缓存依然可以使用

媒体类型的指定

请求数据与媒体类型Content-Type与Accept

同源策略和跨域资源共享

Origin与Access-Control-Allow-Origin
预先使用OPTIONS方法
认证时使用Access-Control-Allow-Credentials: true

定义私有的HTTP首部

一般以x-作为前缀

使用HTTPS加密通信

防范恶意访问的对策

篡改参数获取没有权限的资源

应用大规模访问的对策

限制每个用户的访问，超出上限值的情况响应429 Too Many Requests

向用户告知访问限制的信息

  • X-RateLimit-Limit 单位时间访问上限
  • X-RateLimit-Remaining 剩余的访问次数
  • X-RateLimit-Reset 访问次数重置的时间

附录 B Web API 确认清单

  • URI是否短小且容易输入
  • URI是否能让人一眼看懂
  • URI是否只有小写字母组成
  • URI是否容易修改
  • URI是否反映了服务端架构
  • URI规则是否统一
  • 有没有使用合适的HTTP方法
  • URI里用到的单词所表示的意思是否和大部分API相同
  • URI里用到的名词是否采用了复数形式
  • URI里有没有空格及需要编码的字符
  • URI里的单词和单词之间有没有使用连接符
  • 分页的设计是否恰当
  • 登录有没有使用OAuth 2.0
  • 响应数据格式有没有使用JSON作为默认的格式
  • 是否支持通过查询参数来指定数据格式
  • 是否支持不必要的JSONP
  • 响应数据的内容能不能从客户端指定
  • 响应数据中是否存在不必要的封装
  • 响应数据的结构有没有尽量做到扁平化
  • 响应数据有没有用对像来描述，而不是用数组
  • 响应数据的名称所选用的单词的意思是不是和大部分API相同
  • 响应数据的名称有没有用尽可能少的单词来描述
  • 响应数据的名称由多个单词连接而成时，连接方式在整个API里是否一致
  • 响应数据的名称有没有使用奇怪的缩写形式
  • 响应数据的名称的单复数形式是否和数据内容相一致
  • 出错时响应数据中是不是包含有助于客户端剖析原因的信息
  • 出错时有没有返回HTML数据
  • 有没有返回合适的状态码
  • 服务器端在维护时有没有返回503状态码
  • 有没有返回合适的媒体类型
  • 必要时能不能支持CORS
  • 有没有返回Cache-Control,Etag, Last-Modified,Vary等首部以便客户端采用合适的缓存策略
  • 不想缓存的数据有没有添加Cache-Control: no-cache首部信息
  • 有没有对API进行版本管理
  • API版本的命名有没有遵循语义化的版本控制规范
  • 有没有在URI里嵌入主版本编号，并且能够让人一目了然
  • 有没有考虑API终止提供时的相关事项
  • 有没有在文档里明确注明API的最低提供期限
  • 有没有使用HTTPS来提供API
  • 有没有认真执行JSON转义
  • 能不能识别X-Requested-With首部，让浏览器无法通过SCRIPT元素读取JSON数据
  • 通过浏览器访问的API有没有使用XSRF token
  • API在接受参数时有没有仔细检查非法的参数（负数等）
  • 有没有做到即使请求重复发送，数据也不会多次更新
  • 有没有在响应消息里添加各种增强安全性的首部
  • 有没有实施访问限速
  • 对预想的用例来说限速的次数有没有设置得过少
