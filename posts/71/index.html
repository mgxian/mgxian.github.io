代码整洁之道笔记

发表于 2022-08-15 | 分类于 开发

整洁代码

  • 代码整洁重要性
  • 让营地比你来时更干净

有意义的命名

  • 名副其实
    • 命名宁愿长也要能准确表达意义，不要贪图使用简短命名
  • 避免误导
    • 不要把已经存在公认的单词表示为其他意义
    • 不要使用accountList命名，除非它真是一个List
    • 尽量不要单独使用零和壹（0,O,1,l）
  • 做有意义的区分
    • 如money与moneyAmount没有区别，customer与customerInfo没有区别
  • 使用读得出来的名称
  • 使用可搜索的名称
    • MAX_CLASSES_PER_STUDENT比7更好搜索
  • 避免使用编码
    • 不要在变量名上加入类型名
    • 不要使用变量前缀（m_），定义为类的成员变量
    • 不要给接口前面加上I前缀
  • 避免思维映射
    • 明确是王道
  • 类名
    • 避免使用manager,info,data,processor这样的类名，类名应该是名词，不应该是动词
  • 方法名
    • 方法名应该是动词或者动词短语，修改获取断言应该加上get,set,is,has前缀
  • 别扮可爱
    • 不要使用俚语，特殊语
  • 每个概念对应一个词
    • 例如get,retrieve,fetch只使用一种
    • 例如不要混合使用manager,controller,driver
  • 别用双关语
    • 比如add方法是两个值相加获取新值，如果有个方法把单个参数放到群集（collection），这个命名就不适合，应该叫insert或者append
  • 使用解决方案领域名称
  • 使用源自所涉问题领域的名称
  • 添加有意义的语境
    • 例如当单独使用firstName,lastName,street,houseNumber,city,state,zipcode表示地址信息时，不如使用addrFirstName,addrLastName，更好的方案是创建一个Address类
  • 不要添加没用的语境
    • 只要短名称足够清楚，就要比长名称好，别给名称添加不必要的语境

函数

  • 短小
    • 20行封顶最佳
    • if,else,while语句等，其中的代码块应该只有一句，大抵应该是一个函数调用
  • 只做一件事
    • 如果函数只是做了该函数名下同一抽象层上的步骤，那函数就是只做了一件事
  • 每个函数一个抽象层级
    • 函数中的每一个语句都应该处于同一个抽象层级
  • switch语句
    • 无可避免函数要做多件事，尽量放在较低的抽象层，比如工厂设计模式
  • 使用描述性的名称
    • 函数名宁可长也要描述清楚
    • 如果很难给函数命名，那可能是因为函数做了不止一件事
  • 函数参数
    • 尽量少的参数
    • 参数尽量不要有bool类型的标识值（基于该值做不同分支的操作）
    • 参数如果过多可以使用类传递
    • 一元函数的名称和参数应当形成一种非常良好的动词-名词对形式（writeField(name)）
  • 无副作用
    • 不要做函数名称描述之外的其他事情
  • 分离指令与询问
    • 函数要么做什么事情，要么回答什么事情，二者不可兼得
  • 使用异常代码错误码
    • 抽离try/catch代码块，把它们单独放在一个函数中
    • 错误处理就是一件事情
  • 别重复自己
  • 结构化编程
    • 只要保持函数短小，偶尔出现的return,break,continue语句没有坏处
    • 每个函数尽量减少return,break,continue语句
  • 如何写出这样的函数
    • 先写，然后慢慢改进

注释

  • 注释是在弥补我们用代码表达意图时的失败
  • 注释不能美化糟糕代码
  • 用代码来注释
  • 好注释
    • 法律信息
    • 提供信息的注释
    • 对意图的解释
    • 阐释
    • 警示
    • TODO注释
    • 放大强调看起来不合理之物的重要性
  • 坏注释
    • 喃喃自语
    • 多余的注释
    • 误导性的注释
    • 循规式死板的注释
    • 变更日志式注释
      • 这部分信息应该被存储在源码管理系统里
    • 废话注释
    • 可怕的废话
    • 能用变量和函数就别用注释
    • 括号后面的注释
      • 用来标记代码结构
    • 位置标记
    • 归属与签名
      • 这部分信息应该被存储在源码管理系统里
    • 注释掉的代码
    • HTML注释

格式

  • 格式的目的
    • 更好的可读性
  • 垂直格式
    • 单个文件最好是200-500行左右
    • 注意留空格行
    • 被调用的函数应该放在调用函数的下面
  • 横向格式
    • 最宽不要超过120个字符
    • 水平对齐
    • 缩进
  • 团队规则
    • 整个团队形成统一 规则

对象和数据结构

  • 数据抽象
    • 类里只有数据没有方法
  • 数据、对象的反对称性
    • 过程式（使用数据结构）代码，便于在不改变现在数据结构的情况下添加新函数
    • 面向对象编程便于在不改动现有函数的情况下添加新类
  • 得墨忒耳律
    • 模块不应了解它所操作对象的内部情形
    • 对象隐藏数据，暴露方法
  • 数据传送对象（DTO）
    • 只有变量没有函数的类

错误处理

  • 使用异常而非返回码
  • 先写try-catch-finally语句
  • 尽量不要使用不可控异常（JAVA）
  • 给出异常发生的环境说明
  • 不要返回null值
  • 不要传递null值

边界

  • 使用第三方代码
    • 尽量封装一层
  • 使用尚不存在的代码
    • 提前定义好接口
  • 使用第三方API
    • 尽量封装一层

单元测试

  • TDD三定律
    • 在编写不能通过的单元测试前，不可编写生产代码
    • 只可编写刚好无法通过的单元测试，不能编译也算不通过
    • 只可编写刚好足以通过当前失败测试的生产代码
  • 保持测试代码整洁
  • 尽量少的断言在一个单元测试中
  • 不要和正常代码使用双重整洁标准
  • FIRST原则
    • Fast 测试应该够快
    • Independent 测试应该相互独立
    • Repeatable 测试应该可在任何环境重复通过
    • Self-Validating 测试应该有布尔值输出，无论是通过或者失败，你不应该查看日志文件来确认测试是否通过
    • Timely 测试应该及时编写，单元测试在生产代码之前编写，如果在编写生产代码之后编写测试，你会发现生产代码难以测试

类

  • 类应该短小
    • 单一职责
    • 高内聚
    • 将较大的函数切割为小函数，就会出现更多的类，把小函数的参数变为类的实体变量
  • 为了修改而组织
    • 修改添加特性时，尽量少的修改原来的代码

系统

  • 将系统的构造与使用分开
    • 工厂
    • 依赖注入
  • 方便扩容

迭进

  • 通过迭进设计达到整洁目的
  • 运行所有测试
  • 重构
  • 不可重复
  • 尽量少的类和方法

并发编程

  • 保持同步区的微小
  • 警惕同步方法之间的依赖

逐步改进
